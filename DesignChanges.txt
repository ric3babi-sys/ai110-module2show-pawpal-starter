MISSING RELATIONSHIPS (Data Consistency Issues)
================================================================================

1. BIDIRECTIONAL OWNER-PET RELATIONSHIP
   Issue: When setPet() adds a Pet or when Pet is created, no bidirectional sync occurs
   Impact: Pet objects can exist without being registered in Owner.petList
   Scenario: Owner.setPet(pet) adds pet to list, but Pet.__post_init__() doesn't notify Owner
   Risk: Data inconsistency - "Does this owner have this pet?" question becomes ambiguous
   Location: Owner.setPet() and Pet.__post_init__()
   Fix: Auto-add Pet to owner.petList in Pet.__post_init__() OR validate in setPet()

2. BIDIRECTIONAL PET-TASK RELATIONSHIP
   Issue: When addPetTask() adds a Task, no validation that task.pet matches this Pet
   Impact: Same task object could be added to multiple pets' task lists
   Scenario: task = Task(pet_A, code=0); pet_B.addPetTask(task) - task now in two pets' lists!
   Risk: Task completion affects wrong pet, confusing ownership/responsibility
   Location: Pet.addPetTask()
   Fix: Validate task.pet == self before adding; reject if mismatch

3. TASK-SCHEDULER ONE-WAY RELATIONSHIP
   Issue: Task has no reference to Scheduler objects; Scheduler references Task only
   Impact: Cannot query "What times is this task scheduled?" without external iteration
   Scenario: Need to find all schedulers for task X → must search ALL schedulers in system
   Risk: Performance degradation with large schedules; no history tracking of when task was scheduled
   Location: Task class missing scheduleList field
   Fix: Add List[Scheduler] field to Task; maintain bidirectional link when Scheduler created

4. PET-SCHEDULER RELATIONSHIP BROKEN
   Issue: No way to access Scheduler objects from Pet class directly
   Impact: UI cannot efficiently show "When is this pet's walk scheduled?" without chaining through tasks
   Scenario: User clicks Pet → wants to see all upcoming schedules → must iterate Pet.petTaskList → then...
              Task has no scheduler references, so must search globally
   Risk: Complex query logic scattered across codebase; opportunity for bugs
   Location: Pet class; Scheduler creation
   Fix: Add method getPetSchedulers() or maintain Pet-level scheduler index

5. MISSING PET REMOVAL METHOD
   Issue: Owner class has setPet() to add but no removePet() to delete
   Impact: Cannot unregister a pet; data accumulates forever
   Scenario: Pet is deleted from system but stays in owner.petList
   Risk: Memory leak; incorrect pet counts; orphaned data
   Location: Owner class
   Fix: Add removePet(pet: Pet) -> bool method

CRITICAL BOTTLENECKS (Performance and Logic Issues)
================================================================================

BOTTLENECK #1: LINEAR SEARCH IN getPetByName()
─────────────────────────────────────────────
Location: Owner.getPetByName()
Algorithm: Linear search O(n)
Problem: Every name lookup iterates entire petList
Impact: With 100 pets, 100 lookups = 10,000 list iterations
Scale Issue: Grows linearly; unusable for owners with many pets
Frequency: Likely called frequently by UI to validate pet names
Fix Options:
  • Maintain internal dict {petName -> Pet} for O(1) lookup
  • Use Pet.name as unique key if enforced
  • Implement Least-Recently-Used (LRU) cache for popular lookups

BOTTLENECK #2: NO DUPLICATE PREVENTION
──────────────────────────────────────
Location: Owner.setPet() and Pet.addPetTask()
Problem: Same object can be added multiple times to lists
Impact: Duplicate entries cause incorrect business logic (pet listed twice, task counted twice)
Scenario: owner.setPet(pet); owner.setPet(pet); → now in list twice
          pet.addPetTask(task); pet.addPetTask(task); → task completed twice?
Risk: Duplicate completion notifications, incorrect billing/scheduling, confusion in UI
Fix: Check if already in list before adding: if pet not in self.petList: self.petList.append(pet)

BOTTLENECK #3: NO CASCADING CLEANUP
────────────────────────────────────
Location: Pet.removePetTask() and Owner remove operations
Problem: Removing a Pet/Task doesn't clean up dependent objects
Scenario 1: owner removes pet → Pet's 20 tasks still exist in system (orphaned)
Scenario 2: pet.removePetTask(task) → Task's Scheduler records still exist (orphaned)
Impact: Database/memory bloat; ghost objects; corrupt cascades
Risk: Referential integrity violations; queries return dead references
Example: Scheduler references Task; Task is deleted; Scheduler.getTask() returns None
Fix: Implement cascading delete:
  • Owner.removePet(pet) → delete pet.petTaskList → delete all related Schedulers
  • Pet.removePetTask(task) → delete all Schedulers for that task

BOTTLENECK #4: NO VALIDATION ON NUMERIC FIELDS
───────────────────────────────────────────────
Location: Task.__post_init__(), Scheduler setters
Fields: taskCode, duration, month, date, year, time
Problem: Invalid values accepted without error
Impact Examples:
  • taskCode = 999 (valid codes are 0-3)
  • duration = -30 (negative duration makes no sense)
  • month = 13 (invalid month)
  • date = 99 (invalid date)
  • time = "25:99" (invalid time format)
  • year = 1800 (scheduling for past?)
Risk: Garbage data stored; logic breaks downstream; invalid schedules accepted
Fix: Add validation in __post_init__() and setters:
  • Task: assert 0 <= taskCode <= 3
  • Task: assert duration > 0
  • Scheduler: assert 1 <= month <= 12
  • Scheduler: assert 1 <= date <= 31 (or validate per month/year)
  • Scheduler: assert time matches regex "^\d{2}:\d{2}$" and 0-23:0-59 range

BOTTLENECK #5: NON-THREAD-SAFE ID GENERATION
──────────────────────────────────────────────
Location: Global counters (_owner_id_counter, _pet_id_counter, etc.)
Problem: If app runs concurrent requests, two threads may generate same ID
Scenario: Thread A reads counter=5, Thread B reads counter=5, both increment → both create ID=6
Impact: Duplicate IDs; data corruption; loss of uniqueness guarantees
Risk: Hard-to-reproduce race condition bugs; data integrity violation
Fix: Use threading.Lock() around counter increments:
  with _id_lock:
      _owner_id_counter += 1
      self.ownerID = _owner_id_counter

BOTTLENECK #6: NO TASK STATUS FILTERING
────────────────────────────────────────
Location: Pet class
Problem: getPetTaskList() returns all tasks; no filtering by completion status
Impact: UI must iterate and filter manually every time it wants "pending tasks" or "completed tasks"
Scenario: Show "Tasks Due Today" → must loop through 100 tasks, check completion = O(n) every render
Frequency: Likely called multiple times per screen load
Fix: Add convenience methods:
  • getCompletedTasks() -> List[Task]
  • getPendingTasks() -> List[Task]
  • getTasksByStatus(completed: bool) -> List[Task]

ADDITIONAL ISSUE: MISSING TIMESTAMP ON TASK COMPLETION
─────────────────────────────────────────────────────
Location: Task.doTask() and Task.completed field
Problem: Task only tracks boolean completed status, not WHEN it was completed
Impact: Cannot generate completion reports, audit trails, or task history
Scenario: "When did we last walk the dog?" → No answer available
Fix: Add optional field: completed_at: Optional[datetime] = None
     Update doTask() to set: self.completed = True; self.completed_at = datetime.now()
